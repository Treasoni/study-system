# Design: Tool Registry + Collect Skill Refactoring

## 1. Overview

The `collect` skill is hardcoded to depend on `opencli` for search and fetch operations. This creates a single point of failure, wastes available capabilities (WebSearch/WebFetch/MCP), and forces users to install opencli before using the study system.

**Goal**: Create a tool discovery layer that auto-detects available search/fetch tools and enables the collect skill to dynamically select the best available tools.

**Scope**:
- New skill: `tool-registry` — scans environment, discovers tools, writes registry
- Refactor: `collect` skill — reads registry, uses tools by priority, merges curate logic
- Merge: collect + curate into single phase

## 2. Architecture

```

│                    Tool Discovery Layer                  │
├─────────────────────────────────────────────────────────┤
│                                                         │

│  │ 1. Probe built-in tools (WebSearch,     │            │
│  │    WebFetch — always available)          │            │

│  │ 3. Check CLI tools (opencli, etc.)      │            │
│  │ 4. Evaluate capabilities & reliability  │            │
```

## 3. Tool Registry Design

### 3.1 Registry File Format

```yaml
# ~/.claude/rules/tool-registry.yaml
# Auto-generated by tool-registry skill. Do not edit manually.

metadata:
  generated_at: "2026-06-01T10:00:00Z"
  generator: tool-registry skill v1.0
  ttl_days: 7

search_tools:
  - name: WebSearch
    type: builtin
    priority: 1  # dynamically calculated
    status: available  # available | not_installed | error | disabled
    description: "Claude built-in web search"
    capabilities: [general_search, news, realtime]
    reliability: high  # high | medium | low

  - name: opencli
    type: cli
    priority: 2
    status: available
    description: "opencli search command"
    capabilities: [general_search, deep_content]
    command: "opencli search '{query}' --format json"
    reliability: medium

  - name: acgithub
    type: mcp
    server: acgithub
    priority: 3
    status: available
    description: "GitHub code and repository search"
    capabilities: [code_search, repos, documentation]
    methods: [search_code, search_repositories]
    reliability: high

fetch_tools:
  - name: WebFetch
    type: builtin
    priority: 1
    status: available
    description: "Claude built-in page fetcher"
    capabilities: [url_fetch, text_extraction]
    reliability: high

  - name: opencli
    type: cli
    priority: 2
    status: available
    description: "opencli browser automation"
    capabilities: [browser_fetch, js_rendering, spa_support]
    command: "opencli web get '{url}' --format json"
    reliability: medium
```

### 3.2 Priority Calculation

Priority is **not hardcoded** in the collect skill. It is calculated by tool-registry during scan based on:

1. **Tool availability** — is it installed/connected?
2. **Capability coverage** — how many search/fetch scenarios does it support?
3. **Reliability** — known timeout/failure patterns
4. **User preference** — if user has manually set priority in registry, respect it

The collect skill reads priority from registry and sorts by it. No hardcoded tool order.

### 3.3 Auto-Refresh Logic

```
if registry not exists:
    run tool-registry scan
elif registry.age > 7 days:
    run tool-registry scan
else:
    use existing registry
```

## 4. Collect Skill Refactored Design

### 4.1 New Flow

```
collect skill receives: { topic }

Step 1: Read/refresh registry
  - Check ~/.claude/rules/tool-registry.yaml
  - Auto-refresh if stale (>7 days) or missing

Step 2: Discuss scope with user (BLOCKING)
  - Source count: few (3-5) / medium (8-12) / many (15+)
  - Source types: technical docs / blog posts / videos / forum discussions
  - Depth: quick overview / standard research / deep investigation
  - User's choice determines search strategy and query count

Step 3: Search
  - For each source type, select search tool by registry priority
  - Execute searches with topic + source type as query
  - Collect URLs from results (limited by user's source count choice)

Step 4: Fetch
  - For each URL, select fetch tool by registry priority
  - Fetch content, handle failures gracefully

Step 5: Curate inline (merged from curate skill)
  - Score each file (1-5) on: authority, freshness, completeness, readability
  - Mark duplicates for removal
  - Classify into: core_concepts, practical_examples, advanced
  - Remove low-quality files

Step 6: Output
  - Save curated files to 0-inbox/{topic}/
  - Save metadata.yaml with scores, classifications, and tool usage stats
```

### 4.2 Tool Selection (Dynamic)

The collect skill does NOT hardcode tool order. It reads from registry:

```python
# Pseudocode
registry = read_registry()
search_tools = registry.search_tools
    .filter(status=available)
    .sort(by=priority)

# Use tools in priority order, fallback on failure
for tool in search_tools:
    try:
        results = tool.search(query)
        break
    except:
        continue
```

### 4.3 User Scope Discussion

Before executing searches, collect asks the user:

```
🔍 收集范围确认

主题: {topic}

1. 来源数量？
   ○ 少量 (3-5 个) — 快速概览
   ○ 适中 (8-12 个) — 标准研究
   ○ 大量 (15+ 个) — 深度调研

2. 来源类型？（可多选）
   ☑ 技术文档 (官方文档、API docs)
   ☑ 博客文章 (技术博客、Medium)
   ☐ 视频教程 (YouTube、B站)
   ☐ 论坛讨论 (Stack Overflow、Reddit)

3. 研究深度？
   ○ 快速概览 — 只看核心概念
   ○ 标准研究 — 核心概念 + 实践示例
   ○ 深度调研 — 全面覆盖 + 高级主题
```

User's choices control:
- How many searches to execute
- How many URLs to fetch
- How aggressively to curate

### 4.4 Fallback Strategy

```
Search fallback chain:
  1. Highest priority available tool from registry
  2. Next priority tool if first fails
  3. All tools failed → log error, ask user for manual URLs

Fetch fallback chain:
  1. Highest priority available tool from registry
  2. Next priority tool if first fails
  3. All tools failed → skip URL, log to metadata

If ALL tools unavailable:
  → Log to error-log.md
  → Ask user to provide URLs or install tools
```

## 5. Performance Impact

| Metric | Before | After | Savings |
|--------|--------|-------|---------|
| Subagent calls | 4 (collect, curate, write, beautify) | 3 (collect+curate, write, beautify) | 1 subagent |
| Token overhead | ~15-20k per session | ~10-14k per session | ~5-8k tokens |
| Time | ~2-4 min | ~1.5-3 min | ~30-60s |
| Tool flexibility | opencli only | 3+ tools (dynamic) | 3x tools |
| User control | None (auto-collect all) | Scope discussion before collect | Better UX |

## 6. Testing Strategy

1. **tool-registry skill**: Run with/without opencli, with/without MCP servers, verify correct YAML output
2. **collect skill (no opencli)**: Verify WebSearch/WebFetch fallback works
3. **collect skill (with opencli)**: Verify opencli used when available
4. **collect + curate merge**: Verify scoring, dedup, classification work inline
5. **User scope discussion**: Verify source count limits are respected
6. **Auto-refresh**: Verify registry refreshes after 7 days

## 7. Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Tool-registry scan takes too long | Delay before collect | Cache results, only re-scan when stale |
| Registry file gets corrupted | Collect fails | Validate YAML on read, fallback to defaults |
| User cancels scope discussion | Collect blocked | Use sensible defaults if user skips |
| WebSearch/WebFetch rate limited | Search fails | Fallback to next tool, log error |
| New MCP tool added but not detected | Missed capability | Registry format is extensible, scan logic is pluggable |
